this guide is for library contributors working on aethellib internals.
it describes the current 0.5.1 generator architecture in implementation order, not end-user usage.

## 1. module intent

`src/generator/mod.rs` defines the generation runtime model:

- lifecycle contract (`Generation` trait)
- field abstraction (`GeneratedField<T>`) with owned candidate state
- provenance metadata (`SourceRef`)
- candidate/index primitives (`GeneratedFieldCandidates<T>`, `ProvenanceCandidateIndex<T>`)
- generation errors (`GenerationError`)

`src/generator/utils.rs` contains utility helpers and sampling internals:

- `generated_field_builder(...)`
- `sample_generated_field(...)`

## 2. high-level data flow

current flow from source docs to runtime value:

1. `Generation::compile(corpus)` is called.
2. each output `GeneratedField<T>` is created with `GeneratedField::pending(...)`.
3. candidates are built with `generated_field_builder(...)`.
4. each field receives its compiled state via `set_candidates(field_name, candidates)`.
5. `compile(corpus)` validates required fields while constructing `Self`.
6. runtime generation calls field-level `regenerate_*` methods.
7. each sample returns value + source refs.

important: candidate ownership is field-local by design.
this enables selective re-sampling without external candidate wiring.

## 3. lifecycle contract (`Generation`)

`Generation` intentionally keeps a narrow required surface:

- associated types:
  - `Loader: TargetedLoader`
  - `CompiledState`
  - `Error: std::error::Error + From<MergerError>`
- required methods:
  - `compile(corpus: AethelCorpus<Self::Loader>) -> Result<Self, Self::Error>`
- default constructors:
  - `from_documents(...) -> Result<Self, Self::Error>`
  - `from_files(...) -> Result<Self, Self::Error>`

design note:
- `Generation` does not require `generate()` methods.
- generation execution is currently expressed as inherent methods on implementation structs.

## 4. error model (`GenerationError`)

`GenerationError` variants represent compile/sample failures:

- `NotCompiled`
- `EmptyCandidates { field }`
- `BuilderDefinition { field, reason }`

guideline:
- use `BuilderDefinition` for static setup issues in compile/setup paths.
- use `NotCompiled` when runtime sampling is invoked without attached candidates.
- use `EmptyCandidates` when candidate arrays are empty at sample time.

## 5. provenance primitives

### 5.1 `SourceRef`

`SourceRef` is the field-level provenance atom:

- `source_id`
- `source_name`
- `section`
- `field`

`matches(section, field)` is a convenience predicate.

### 5.2 `ProvenanceCandidateIndex<T>`

maps candidate value -> source refs:

- `insert(value, source_ref)`
- `refs_for(&value)`
- `contains_value(&value)`

invariant:
- for each value key, source refs are de-duplicated with first-seen ordering.

## 6. field abstraction (`GeneratedField<T>`)

`GeneratedField<T>` contains both output and local generation state:

- sampled output:
  - `value`
  - `source_refs`
- attached compile state for self-managed re-sampling:
  - `field_name: Option<String>`
  - `compiled_candidates: Option<GeneratedFieldCandidates<T>>`

public responsibilities:

- accessors:
  - `value()`, `source_refs()`, `source_ids()`, `source_paths_in(...)`
- compile-state management:
  - `set_candidates(...)`, `is_compiled()`
- sampling control:
  - `regenerate_with_rng(...)`, `regenerate(...)`
  - `regenerate_in_place_with_rng(...)`, `regenerate_in_place(...)`

critical behaviour:
- `regenerate_*` fails with `NotCompiled` if field candidates are not attached.
- `GeneratedFieldCandidates::sample(...)` re-attaches candidate state to the returned field, preserving future re-sampling capability.

## 7. candidate container (`GeneratedFieldCandidates<T>`)

`GeneratedFieldCandidates<T>` holds:

- `values: Vec<T>`
- `provenance: ProvenanceCandidateIndex<T>`

sampling API:
- `sample(field_name, rng)` returns `GeneratedField<T>`.

sampling responsibilities:
- produce a sampled `value`.
- attach source refs for that value.
- map empty candidate failures to field-scoped errors.
- attach candidate state into returned field.

## 8. candidate utilities (`utils.rs`)

`generated_field_builder(corpus, section, field, extract_values)` returns `GeneratedFieldCandidates<T>` directly:

- iterates source documents
- extracts values via closure
- appends values to candidate vector
- inserts value -> `SourceRef` into provenance index

guideline:
- use candidate helpers in `Generation::compile(...)` to keep extraction logic local and explicit.

## 9. implementation recipe for new generation types

when adding a new internal generation type:

1. define struct fields as `GeneratedField<T>`.
2. in `compile(corpus)`:
   - create each field with `GeneratedField::pending(...)`.
  - create candidates with `generated_field_builder(...)`.
   - attach candidates with `set_candidates(...)`.
3. still inside `compile(corpus)`:
  - verify every required field with `is_compiled()`.
4. implement inherent `generate_with_rng` and/or `generate`:
   - call per-field `regenerate_*` methods.
   - support selective reroll methods for lockable fields.

## 10. invariants to preserve

do not break these without coordinated migration:

- re-sampled fields must retain attached candidates for future rerolls.
- source ref de-duplication order is stable (first-seen).
- errors should remain field-scoped where possible.
- field-level resampling must not require external candidate passing.

## 11. known limitations (0.5.1)

- `Generation::CompiledState` is present but not strongly integrated into the runtime shape.
- no built-in helper yet for aggregated provenance across composed fields (for example full-name composition).
- no module-level unit tests for generator internals yet; behaviour is mostly example-driven.

## 12. recommended next internal tasks

1. add dedicated unit tests under `src/generator` for:
   - `GeneratedField::regenerate_*`
   - candidate attachment persistence
2. add composed-provenance helper APIs for multi-field derived strings.
3. decide whether `Generation::CompiledState` should become an enforced field-level abstraction or be removed.

## 13. file map

- core contract/types: `src/generator/mod.rs`
- build and sample helpers: `src/generator/utils.rs`
- expanded reference example: `examples/generation_0_5.rs`
